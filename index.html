Unit 1

Introduction, SRS & Requirements

Q1
Define Software Engineering. Explain its objectives, goals, evolution, role in modern computing, software services, software complexity and principles used to manage complexity.
+

Definition

Software Engineering is the systematic, disciplined and quantifiable application of engineering principles to the design, development, operation and maintenance of software. The classic IEEE definition: "the application of a systematic, disciplined, quantifiable approach to the development, operation and maintenance of software."

In simple words, it is the art of building reliable, efficient and maintainable software in a cost-effective way using well-defined methods, instead of writing code in an ad-hoc manner.

Objectives & Goals

  • Correctness: software must meet its requirements.
  • Reliability: it should work consistently without failures.
  • Efficiency: optimum use of time, memory and resources.
  • Maintainability: easy to modify, fix and enhance.
  • Usability: easy to learn and use.
  • Cost-effectiveness: deliver within budget and schedule.
  • Portability & Reusability: usable across platforms and reusable in future projects.

Evolution of Software Engineering

  1. Early period (1950s): programs written for specific machines, no methodology.
  2. Control-flow / exploratory style (1960s): code grew large and unmanageable — led to the software crisis.
  3. Structured / data-flow oriented design (1970s): introduction of structured programming, modularization, SDLC.
  4. Object-oriented design (1980s–90s): objects, classes, reuse, UML.
  5. Modern era (2000s onward): Agile, DevOps, cloud, component-based and service-oriented development.

Role in Modern Computing & Software Services

Software engineering powers almost every domain — banking, healthcare, e-commerce, telecom, AI and embedded systems. Software services (like SaaS, web services, APIs and cloud computing) deliver software functionality over the internet on demand, making software engineering central to digital infrastructure.

Software Complexity

Software is inherently complex because of: large size, invisible structure, frequent changes, many interacting components, and lack of physical constraints. Complexity makes software hard to understand, test and maintain.

Principles to Manage Complexity

PrincipleMeaning
AbstractionFocus on essential details, hide the rest.
Decomposition / ModularizationBreak a big problem into smaller manageable modules.
HierarchyOrganize modules in layers/levels.
Information hidingHide internal implementation behind interfaces.
ReuseUse existing tested components.
Key PointTwo principles control complexity: Abstraction (ignore unnecessary detail) and Decomposition (divide and conquer).
Exam TipAlways start with the IEEE definition, then list goals as a bullet list, and finish with abstraction + decomposition for complexity — examiners look for those keywords.
Q2
What is Software Crisis? Discuss its causes, impacts and how Software Engineering helps overcome it.
+

Definition

The Software Crisis (term coined in 1968) refers to the situation in early computing when software projects routinely exceeded budgets, missed deadlines, were unreliable, and were hard to maintain — because hardware power was growing faster than the ability to produce good software.

Causes

  • Rapid increase in software size and complexity.
  • Lack of proper methodologies and tools.
  • Poor or changing requirements.
  • Inadequate documentation and no project management.
  • Shortage of skilled developers.

Impacts

  • Projects went over budget and over schedule.
  • Software was low quality, buggy and unreliable.
  • Difficult and costly maintenance.
  • Many projects were cancelled or never delivered.

How Software Engineering Helps Overcome It

  1. Use of SDLC models (Waterfall, Iterative, Agile) for structured development.
  2. Formal requirement engineering (SRS).
  3. Systematic design, coding standards and modularization.
  4. Rigorous testing, verification and validation.
  5. Proper project management, estimation (COCOMO) and documentation.
  6. CASE tools and reusable components.
NoteThe software crisis is the historical reason software engineering was born as a discipline.
Q3
Explain Software Scope and its importance in software projects.
+

Definition

Software Scope defines the boundaries of the software — what features, functions, data and behaviour the system will (and will not) include. It answers: "What exactly are we building?"

Software scope describes

  • Functions and features to be delivered.
  • Inputs, outputs and data handled.
  • Performance, constraints and interfaces.
  • What is excluded (out of scope).

Importance

  1. Provides a clear understanding of deliverables to all stakeholders.
  2. Basis for cost, effort and time estimation.
  3. Prevents scope creep (uncontrolled growth of requirements).
  4. Helps in planning, scheduling and resource allocation.
  5. Defines the acceptance criteria for the final product.
Exam TipLink scope to estimation and scope creep — those two terms usually fetch marks.
Q4
What is SRS? Discuss its importance, characteristics of a good SRS, categories of users, why SRS is a black-box specification, and how it resolves incomplete and ambiguous requirements.
+

Definition

A Software Requirement Specification (SRS) is a formal document that completely describes the functional and non-functional requirements of the software to be built. It is the agreement (contract) between the customer and the developer.

Importance of SRS

  • Acts as a contract / agreement between client and developer.
  • Forms the basis for design, coding and testing.
  • Reduces development effort and misunderstandings.
  • Helps in cost and schedule estimation.
  • Provides a baseline for validation (verify the product against SRS).

Characteristics of a Good SRS

CharacteristicMeaning
CorrectEvery requirement is genuinely needed.
CompleteAll requirements are stated.
UnambiguousEach requirement has only one interpretation.
ConsistentNo conflicting requirements.
VerifiableCan be tested/checked.
ModifiableEasy to change without side effects.
TraceableEach requirement can be traced to design/code.
RankedRequirements prioritized by importance/stability.

Categories of Users of SRS

  • Customers/Clients: to confirm requirements are correct.
  • Users: to understand what the system will do.
  • Developers/Designers: to design and build the system.
  • Testers: to create test cases.
  • Project managers: for planning and estimation.
  • Maintenance engineers: for future modifications.

Why SRS is a "Black-Box" Specification

SRS describes what the system should do (external behaviour: inputs → outputs) without specifying how it is implemented internally. The internals are hidden like a black box.

Example: For a calculator, SRS states "Given two numbers and operator '+', output their sum." It does not state which algorithm or data structure performs the addition.

Resolving Incomplete & Ambiguous Requirements

  1. Incomplete: through repeated requirement elicitation, interviews, prototyping and reviews, missing requirements are discovered and added.
  2. Ambiguous: by using precise language, decision tables, diagrams, examples and formal notations, each requirement is given a single clear meaning.
  3. Stakeholder review and sign-off validates that requirements are complete and clear.
Key PointSRS = WHAT, not HOW. That single line explains the black-box nature.
Q5
Explain Requirement Gathering and Requirement Analysis. Discuss gathering techniques and classify Functional vs Non-Functional requirements with examples (Banking System).
+

Requirement Gathering (Elicitation)

The process of collecting requirements from stakeholders (customers, users, domain experts) to understand what the system should do.

Requirement Analysis

The process of studying, refining and organizing the gathered requirements — removing conflicts, ambiguity and incompleteness — and modelling them clearly before documenting in the SRS.

Requirement Gathering Techniques

  • Interviews: direct questioning of stakeholders.
  • Questionnaires/Surveys: collect info from many users.
  • Observation: watching users perform current tasks.
  • Brainstorming: group idea generation.
  • Prototyping: build a sample to clarify requirements.
  • Document/record review: study existing system documents.
  • Use cases & scenarios: describe system interactions.

Functional vs Non-Functional Requirements

BasisFunctional (FR)Non-Functional (NFR)
MeaningWhat the system does (features/functions)How well the system performs (quality)
FocusBehaviourConstraints/quality attributes
Example typeLogin, transfer moneySpeed, security, reliability

Example — Banking System

Functional Requirements:

  • Customer can log in with ID and password.
  • Customer can withdraw, deposit and transfer money.
  • System generates account statements.
  • System calculates interest.

Non-Functional Requirements:

  • Security: transactions must be encrypted.
  • Performance: a transaction should complete within 2 seconds.
  • Availability: system available 24×7.
  • Reliability: no data loss during failures.
  • Usability: simple, easy-to-use interface.
Exam TipRemember: FR = "functions", NFR = "qualities" (performance, security, usability, reliability, portability).

+

When requirements contain complex logic with many conditions and actions, plain text becomes ambiguous. Decision trees and decision tables represent this logic clearly and completely.

Decision Tree

A tree-like diagram where branches represent conditions and leaves represent actions. It is easy to read and good when conditions follow in a sequence.

Decision Tree — ATM withdrawal logic

Decision Table

A table with condition stubs (top) and action stubs (bottom). Each column is a rule combining conditions and their resulting actions. It guarantees all combinations are covered.

Process Models & Agile

Q7
Explain SDLC and its purpose. Discuss popular SDLC models and methodologies. Which model is best and why?
+

What is SDLC?

The Software Development Life Cycle (SDLC) is a structured sequence of phases used to develop software systematically — from initial idea to delivery and maintenance.

Typical SDLC Phases

Standard phases of the Software Development Life Cycle

Purpose of SDLC

  • Provides a systematic, planned approach to development.
  • Improves quality, control and predictability.
  • Helps in estimation, scheduling and risk management.
  • Ensures the product meets requirements.

Popular SDLC Models / Methodologies

  • Waterfall — linear, sequential.
  • Iterative / Incremental — build in increments.
  • V-Model — verification & validation focused.
  • Prototyping — build sample first.
  • Spiral — risk-driven.
  • Evolutionary — evolve through versions.
  • Agile / Scrum / XP — adaptive, iterative, customer-focused.
Which is best?There is no single best model — it depends on the project. Agile is best for changing requirements and frequent delivery; Waterfall suits small, stable, well-understood projects; Spiral suits large, high-risk projects.
Exam TipNever say one model is universally best. Always answer: "best depends on requirements stability, project size and risk."
Q8
Explain the Classical Waterfall Model, Iterative Waterfall Model and Phased Development Lifecycle — phases, advantages, disadvantages and comparison.
+

Classical (Pure) Waterfall Model

A linear, sequential model where each phase must be fully completed before the next begins, and there is no going back.

Classical Waterfall — phases flow downward like a waterfall

Iterative Waterfall Model

The practical, improved version — same phases but with feedback paths that allow going back to earlier phases to correct errors found later. This is the most widely used basic model.

Phased Development Lifecycle

The product is delivered in phases/parts. The system is divided so that a usable subset is released first and further phases add functionality — combining structured phases with incremental delivery.

Advantages (Waterfall)
  • Simple and easy to understand.
  • Well documented, easy to manage.
  • Good for small, stable projects.
Disadvantages (Waterfall)
  • No working software until late.
  • Poor for changing requirements.
  • Hard to go back (classical).
  • Risk found very late.

Comparison

BasisClassical WaterfallIterative WaterfallPhased Development
FeedbackNonePhase-to-phase feedbackAcross phases/releases
DeliveryOne final productOne final productIn multiple phases
Change handlingVery poorBetterGood
UseTheoretical/idealPractical projectsLarge staged projects

+

V-Model (Verification & Validation Model)

The V-Model is an extension of the waterfall model where every development phase (left side) is associated with a corresponding testing phase (right side). Development goes down, testing goes up, forming a "V".

V-Model — each design phase maps to a testing phase

Suitability

  • Projects with clear, fixed requirements.
  • Small to medium, safety-critical systems (medical, defence).
  • Where strong testing/quality is required.
Advantages
  • Testing planned early (each phase).
  • High quality, fewer defects.
  • Simple and disciplined.
Disadvantages
  • Rigid, poor for changing requirements.
  • No early prototype.
  • Costly to handle late changes.

V-Model vs Incremental Model

Q10
Explain the Incremental Model with its suitability, merits and limitations.
+

Definition

In the Incremental Model, the software is built and delivered in increments (parts). Each increment adds functionality to the previous one, and each goes through requirement, design, code and test. The first increment is usually the core product.

Incremental Model — software grows increment by increment

Suitability

  • Large projects that can be divided into parts.
  • When early partial delivery is needed.
  • Requirements are reasonably known but may be refined.
Merits
  • Working software delivered early.
  • Easier to test and debug smaller parts.
  • Flexible — feedback used in next increment.
  • Lower initial delivery risk.
Limitations
  • Needs good planning and architecture.
  • Total cost may be higher.
  • Not ideal if requirements are completely unclear.
Q11
Explain the Evolutionary Model and its merits and limitations.
+

Definition

The Evolutionary Model develops software through successive versions/builds. An initial version is built quickly, given to users, and then refined and expanded based on feedback until the complete system evolves. It combines iteration with incremental delivery.

Suitability

  • Requirements are not fully known at the start.
  • Large projects where user feedback is valuable.
  • Object-oriented and rapidly changing domains.
Merits
  • Handles changing/unclear requirements well.
  • Early user feedback improves the product.
  • Working versions available early.
Limitations
  • Hard to plan cost and schedule.
  • Poor structure if not managed carefully.
  • Continuous changes may degrade design.
NoteIncremental adds new features; Evolutionary refines/grows the same product through versions.
Q12
Explain the Prototyping Model — its types, applications, benefits, selection criteria, with a suitable case study.
+

Definition

In the Prototyping Model, a working prototype (sample model) of the system is built quickly and shown to the user. Based on feedback, it is refined repeatedly until requirements are clear, then the final product is built.

Prototyping cycle — build, evaluate, refine until requirements freeze

Types of Prototyping

  1. Throwaway / Rapid: prototype is discarded after requirements are clear; final system built fresh.
  2. Evolutionary: the prototype is refined and gradually becomes the final product.
  3. Incremental: system built as several prototypes merged together.
  4. Extreme prototyping: mainly used for web development (static → functional → services).

Applications & Selection Criteria

  • When requirements are unclear/changing.
  • For UI-heavy systems.
  • When user involvement is high.
  • New, innovative or risky projects.
Benefits
  • Clarifies requirements early.
  • Reduces misunderstanding & rework.
  • High user satisfaction.
Drawbacks
  • Can increase cost/time.
  • Users may mistake prototype for final.
  • Poor documentation if rushed.
Case StudyBuilding an online food-ordering app: the team first makes a clickable UI prototype of the menu and cart, shows it to the restaurant owner, gathers feedback (add filters, payment options), refines it 2–3 times, then develops the full product. This avoids building the wrong system.

Agile is an iterative and incremental software development approach that focuses on customer collaboration, working software, flexibility and rapid delivery in short cycles called iterations/sprints.

Agile Manifesto Values

  4. Daily collaboration between business and developers.

  6. Self-organizing teams and regular reflection (retrospectives).

Q14
Explain Scrum Framework, Scrum Lifecycle and Scrum Process Model with neat diagram.
+

What is Scrum?

Scrum is the most popular Agile framework that develops software in fixed-length iterations called Sprints (usually 2–4 weeks), delivering a potentially shippable increment each sprint.

Scrum Roles

  • Product Owner: manages the product backlog, represents the customer.
  • Scrum Master: facilitates the process, removes obstacles.
  • Development Team: builds the increment.

Scrum Artifacts & Events

  • Product Backlog: prioritized list of all requirements.
  • Sprint Backlog: items chosen for the current sprint.
  • Sprint Planning, Daily Scrum (stand-up), Sprint Review, Sprint Retrospective.
Scrum Process Model — backlog → sprint → increment

Scrum Lifecycle (steps)

  1. Product Owner creates the Product Backlog.
  2. Sprint Planning selects items into the Sprint Backlog.
  3. Team works during the Sprint, holding Daily Scrums.
  4. Sprint Review demonstrates the increment.
  5. Sprint Retrospective improves the process; next sprint begins.
Q15
What is Extreme Programming (XP)? Explain its key practices.
+

Definition

Extreme Programming (XP) is an Agile method that emphasizes technical excellence, customer satisfaction and frequent releases through disciplined engineering practices. It is best for projects with rapidly changing requirements.

Key Practices

  • Pair Programming: two developers write code at one workstation.
  • Test-Driven Development (TDD): write tests before code.
  • Continuous Integration: integrate and test code frequently.
  • Refactoring: continuously improve code structure without changing behaviour.
  • Small/Frequent Releases: deliver in short cycles.
  • Simple Design: keep design as simple as possible.
  • On-site Customer: a customer representative is always available.
  • Collective Code Ownership & Coding Standards.
  • Sustainable Pace (40-hour week).

XP Values

Communication, Simplicity, Feedback, Courage, Respect.

Exam TipThe two most-asked XP practices are Pair Programming and Test-Driven Development.
Q16
Which software process model is best suited for risk management? Explain with advantages and disadvantages.
+

Answer: The Spiral Model

The Spiral Model (by Barry Boehm) is the best model for risk management because it is risk-driven — each loop of the spiral explicitly includes risk analysis and resolution before proceeding.

Four Quadrants of Each Spiral Loop

  1. Determine objectives and constraints.
  2. Risk analysis — identify and resolve risks (build prototypes).
  3. Develop & test the next version.
  4. Plan the next iteration.
Spiral Model — risk analysis in every loop
Advantages
  • Strong risk management.
  • Good for large, complex, high-risk projects.
  • Early prototypes; flexible.
Disadvantages
  • Complex and costly.
  • Needs risk-assessment expertise.
  • Not suitable for small projects.

+

Definition

Exploratory style programming (also called the build-and-fix or ad-hoc approach) is an unstructured way of writing code where the programmer directly starts coding based on intuition, without proper requirement analysis, design or documentation, and keeps fixing it by trial and error.

Limitations of Exploratory Style

  • Becomes unmanageable as program size grows.
  • Very hard to understand, debug and maintain.
  • No documentation → knowledge lost.
  • Cost and effort grow exponentially with size.
  • Poor quality, unreliable, error-prone.
  • No reusability or proper structure.

Q18
Explain Structured Analysis and Structured Design. Differentiate between them with examples. Also explain the role of DFD and Structure Charts.
+

Structured Analysis (SA)

A technique to understand and model the problem by analysing the flow and transformation of data in the system. Its main tool is the Data Flow Diagram (DFD). It answers "WHAT" the system does.

Structured Design (SD)

A technique to convert the analysis model into a software structure (modules and their relationships). Its main tool is the Structure Chart. It answers "HOW" the system is organized.

Difference

BasisStructured AnalysisStructured Design
GoalUnderstand the problem (WHAT)Plan the solution (HOW)
Main toolDFD, data dictionaryStructure chart
OutputLogical model of data flowModule hierarchy
FocusData & processesModules & control

Role of DFD & Structure Charts

  • DFD: shows how data flows between processes, stores and external entities — used in analysis.
  • Structure Chart: shows the hierarchy of modules and how control/data passes between them — used in design. The DFD is transformed into a structure chart using transform analysis and transaction analysis.
Structure Chart — module hierarchy with control flow

Q20
What is Modularization? Explain module independence, coupling and cohesion. Discuss their role in software quality and design strategy.
+

Modularization

Modularization is the process of dividing software into separate, manageable modules, each performing a well-defined function. It applies the principle of divide and conquer.

Module Independence

A good design has independent modules — each module can be developed, tested and changed without affecting others. Module independence is measured by two criteria: coupling and cohesion.

Coupling

Coupling = the degree of interdependence between modules. Goal: LOW coupling (modules should be loosely connected).

Cohesion

Cohesion = the degree to which elements within a single module belong together. Goal: HIGH cohesion (a module should do one well-defined task).

Aim for HIGH cohesion within modules and LOW coupling between them

Role in Software Quality & Design Strategy

  • Maintainability: low coupling means changes stay local.
  • Reusability: highly cohesive modules are easy to reuse.
  • Testability: independent modules are easy to test separately.
  • Understandability & Reliability: well-defined modules reduce errors.
Key Point (Golden Rule of Design)A good design has LOW coupling and HIGH cohesion.

Q23
What is Good Software Design? Explain the fundamental principles of software design.
+

Good Software Design

A good software design is one that is correct, efficient, understandable, and easy to maintain. It satisfies all requirements (correct & complete), uses resources efficiently, has low coupling/high cohesion, and is well documented.

Characteristics of a Good Design

  • Correctness — implements all requirements.
  • Understandability — clear and simple structure.
  • Maintainability — easy to modify.
  • Efficiency — good use of resources.

Fundamental Principles of Software Design

  1. Abstraction: focus on essential features, hide details.
  2. Modularity: divide into independent modules.
  3. Information Hiding / Encapsulation: hide internal data behind interfaces.
  4. Stepwise Refinement: develop design top-down, adding detail gradually.
  5. Functional Independence: low coupling, high cohesion.
  6. Hierarchy / Structure: arrange modules in layers.
  7. Refactoring: improve design without changing function.
Exam TipThe 4 most important principles to write are: Abstraction, Modularity, Information Hiding, and Stepwise Refinement.

Equivalence Partitioning

Divide input data into partitions (classes) where the system is expected to behave the same. Test one value from each class instead of all values.

Example: Age field valid 18–60. Classes: invalid (<18), valid (18–60), invalid (>60). Test one value from each (e.g., 10, 30, 70).

Boundary Value Analysis (BVA)

Most errors occur at the boundaries/edges of input ranges, so test values at and around the boundaries.

Example: For range 18–60, test: 17, 18, 19 (lower) and 59, 60, 61 (upper).

Exam TipBoth BVA and Equivalence Partitioning are black-box techniques. BVA tests edges; partitioning tests classes.

Q35
Explain Basis Path Testing, Cyclomatic Complexity, Independent Paths, Drivers and Stubs in Unit Testing.
+

Basis Path Testing

A white-box technique (by McCabe) that derives a set of independent execution paths through the program using its Control Flow Graph (CFG), and designs test cases to cover each path at least once.

Cyclomatic Complexity (V(G))

A metric that measures the number of independent paths in the program — i.e., the minimum number of test cases for full path coverage. Formulas:

  • V(G) = E − N + 2 (E = edges, N = nodes)
  • V(G) = P + 1 (P = number of predicate/decision nodes)
  • V(G) = number of enclosed regions + 1

Independent Path

A path through the program that introduces at least one new edge/statement not covered by previous paths. The number of such paths equals the cyclomatic complexity.

Drivers and Stubs

TermMeaningUsed in
StubDummy called module (replaces lower module)Top-down testing
DriverDummy calling module (replaces upper module)Bottom-up testing
Memory TrickStub = below (called), Driver = above (caller). They are temporary scaffolding used in unit testing.
Q36
Solve CFG, Cyclomatic Complexity and Independent Path problems. Explain McCabe's Cyclomatic Complexity metric and its uses.
+

McCabe's Cyclomatic Complexity

It quantifies the logical complexity of a program from its Control Flow Graph (CFG). Lower V(G) = simpler, easier to test code.

Example CFG (if-else inside a loop)

Sample CFG: Nodes = 5, Edges = 6, Decision nodes = 1 (loop) + 1 (branch)

Solved Calculation (for above CFG)

Let N = 5 nodes, E = 6 edges, P = 2 predicate nodes.

  • V(G) = E − N + 2 = 6 − 5 + 2 = 3
  • V(G) = P + 1 = 2 + 1 = 3
  • Regions + 1 = 2 + 1 = 3 ✓ (all methods agree)

So we need 3 independent paths / test cases. Example independent paths: 1-2-3-5, 1-2-4-5, and the loop-back path 1-2-3-5-2-... .

Uses of Cyclomatic Complexity

  • Tells the minimum number of test cases for path coverage.
  • Measures code complexity and maintainability.
  • Helps identify modules needing refactoring (high V(G) = risky).
  • Aids in effort estimation and quality control.
Key FormulaeV(G) = E − N + 2 = P + 1 = Regions + 1.

COCOMO

COCOMO (Constructive Cost Model), by Barry Boehm, estimates effort, time and cost of a software project based on its size in KLOC (thousands of lines of code).

Quick rough estimate using only size:

2. Intermediate COCOMO

3. Detailed COCOMO

Applies cost drivers at each phase of the SDLC separately for the most accurate estimate.

Solved Numerical (Basic COCOMO)

Problem: An organic project of size 32 KLOC. Find effort and development time.

  • Effort E = 2.4 × (32)^1.05 ≈ 2.4 × 38.3 ≈ 91.9 person-months
  • Time D = 2.5 × (91.9)^0.38 ≈ 2.5 × 5.66 ≈ 14.2 months
  • Average staff = E / D ≈ 91.9 / 14.2 ≈ 6.5 persons

Q38
Explain Software Maintenance, Re-engineering and Reverse Engineering with suitable examples.
+

Software Maintenance

Software maintenance is modifying software after delivery to correct faults, improve performance, or adapt to a changed environment.

Types of Maintenance

TypePurposeExample
CorrectiveFix bugs/errorsFix a login crash
AdaptiveAdapt to new environmentUpdate for new OS/tax law
PerfectiveImprove/enhance featuresAdd a search filter
PreventivePrevent future problemsRefactor risky code

Re-engineering

Software re-engineering = examining and restructuring/rebuilding existing software to improve quality, maintainability or migrate to new technology, without changing its functionality.

Example: Converting an old COBOL banking system into a modern Java system while keeping the same features.

Reverse Engineering

Reverse engineering = analysing existing software (often with little documentation) to recover its design, structure and specifications — going from code → design → requirements (backward direction).

Example: Studying a legacy app's code to draw its DFDs/class diagrams when the original documentation is lost.

Reverse engineering recovers design; re-engineering rebuilds an improved system
NoteReverse engineering = code → design (understand). Forward + reverse = re-engineering (rebuild).
